# Claude Agent Cooperation Demo

Claude Code + tmux を使ったマルチエージェント協調パイプラインの実験的実装。複数の Claude Code インスタンスがそれぞれ独立したロールを持ち、tmux のメッセージパッシングとファイルシステムを介して協調動作する構造を検証する。

---

## アーキテクチャ

```
orchestrator
    │
    ├─ writer     ← サブテーマの記事を執筆
    │     │
    ├─ reviewer   ← 論理整合性・事実確認・必要に応じてウェブ検索
    │     │
    └─ editor     ← 全サブテーマを統合してサマリーを生成
```

各エージェントは専用ディレクトリ内の `CLAUDE.md` に役割と手順が定義されており、それがそのまま仕様書として機能する。エージェント同士は直接通信せず、orchestrator を中継点とすることで制御を一元化している。

---

## 動作の仕組み

### 通信プロトコル

エージェント間の通信は tmux の `send-keys` によるメッセージパッシングで行う。ポーリングは不要で、各エージェントからのメッセージが届いたら orchestrator が即座に次のエージェントへルーティングする。

```
writer   → "writer完了: shared/<テーマ>/<サブテーマ>.md
            メモ: <判断・迷い・情報不足のサマリー>"

reviewer → "reviewer完了: shared/<テーマ>/<サブテーマ>_reviewed.md (修正あり|修正なし)
            メモ: <修正内容・残課題のサマリー>"

editor   → "editor完了: shared/<テーマ>/summary.md"
```

### ファイル構造

実行単位はテーマ名ディレクトリで、複数テーマの成果物が混在しない。

```
shared/
  <テーマ名>/
    <サブテーマ名>.md           # writer が生成
    <サブテーマ名>_reviewed.md  # reviewer が生成
    summary.md                  # editor が生成
```

---

## 起動方法

```bash
./start.sh
```

新しい tmux セッション `agents` が作成され、以下のレイアウトで各エージェントが起動する。

```
┌─────────────────┬─────────────────┐
│                 │    writer       │
│  orchestrator   ├─────────────────┤
│                 │    reviewer     │
│                 ├─────────────────┤
│                 │    editor       │
└─────────────────┴─────────────────┘
```

あとは orchestrator ペインで `CLAUDE.md` の手順に従って指示を出すだけ。

## 設計思想

### 疎結合と制御の維持

エージェント数が増えると peer-to-peer 通信（誰が誰に何を送っても良い）は表現力が高い反面、「誰が何を決めたか」が見えにくくなる。この設計では orchestrator を中継点にすることで、パイプラインのどこで何が起きているかを常に把握できる状態を維持している。これは並列効率を多少犠牲にするが、デバッグのしやすさと制御の明確さを優先した選択である。

### 思考コンテキストの受け渡し

エージェント間で「成果物」だけを渡すと、次のエージェントは前のエージェントが**なぜそう判断したか**を知ることができない。各エージェントの内部推論はコンテキストウィンドウの中に閉じており、ファイルに書き出さない限り他のエージェントには伝わらない。これは peer-to-peer 通信でも同じ制約であり、エージェントが明示的に言語化しない限り思考の経緯は共有されない。

この設計では完了通知にメモを添えることで思考の断片を渡している。writer が「迷った箇所・情報不足」として挙げた部分を reviewer が重点的に検証し、その残課題を editor がサマリーの考察に反映する、という情報の連鎖が生まれる。「成果物だけ渡す」から「成果物＋思考の断片を渡す」への転換は実装コストがほぼゼロである一方、エージェント間の情報の質を大きく改善する。これはエージェント設計というよりコミュニケーション設計の問題であり、人間のチームにも同様に当てはまる原則である。

## Claude Agent Teams との比較

Claude Code には [Agent Teams](https://code.claude.com/docs/ja/agent-teams) という公式のマルチエージェント機能（実験的）が存在する。本リポジトリの構成はそれを自前で実装したものに相当し、動作の仕組みもほぼ同じ設計に収束している。

自前で実装する最大の利点は**透明性**にある。tmux + CLAUDE.md で何が起きているかが全て読めるため、問題が起きたときに原因を特定しやすく、設計の意図が失われない。Agent Teams はセットアップが不要で peer-to-peer 通信や共有タスクリストといった機能を持つが、内部の調整ロジックは自前実装ほど手元で制御できない。

### Agent Teams に移行するなら

自前実装から Agent Teams への移行は、変更箇所が少ない。各エージェントの `CLAUDE.md` を `.claude/agents/` 以下に subagent 定義として配置し、tmux による手動ルーティング（`tmux send-keys ...`）を削除するだけで対応できる。ファイル構造（`shared/<テーマ名>/`）やエージェントの役割定義はそのまま再利用できる。

```
.claude/agents/
  writer.md    ← writer/CLAUDE.md の役割定義部分
  reviewer.md  ← reviewer/CLAUDE.md の役割定義部分
  editor.md    ← editor/CLAUDE.md の役割定義部分
```

変わるのは「配管部分」だけで、パイプラインの設計思想は自前実装でも Agent Teams でも同じである。Agent Teams の仕様が変わっても、ここで得た設計の考え方はそのまま適用できる。
